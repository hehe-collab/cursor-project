# 天马1 印尼成片：`_scratch_liuxing/天马1_印尼结构复制_v2.mp4` 是怎么来的、和「正确」的关系

## 1. 本目录下几个文件的技术对比（ffprobe，2026-03-30 快照）

| 文件 | duration（秒） | 视频 start_time | 视频约码率 |
|------|----------------|-----------------|------------|
| 爆款 `天马1.mp4` | **227.277** | 0 | （略） |
| `天马1_印尼结构复制.mp4`（首版） | 227.281 | 0.021 | ~1.31 Mbps |
| `天马1_印尼结构复制_v2.mp4` | **227.317** | 0.063 | ~2.16 Mbps |
| `天马1_印尼结构复制_v3.mp4` | **227.317** | 0.063 | ~2.17 Mbps |

结论：

- **v2 与 v3**：时长、视频 `start_time`、帧率分母一致，属于**同一类重编码链路**的产物；文件大小只有极小差别（第二次编码 `concat` 或分段码率抖动）。
- **首版 `结构复制.mp4`**：码率明显更低、容器时间戳也与 v2 不同，更像是**另一套参数或曾用流复制/不同拼接方式**得到的版本，与 v2 不是同一条导出链。

## 2. v2「对」的时候，用的核心方法是什么（与仓库脚本一致）

与当前 `match_report.json`（`_scratch_liuxing`）一致的算法是：

1. **全剧中文音轨按集拼接** → 与 **整条爆款音轨** 做互相关（窗口化粗搜 + 窄窗精相关；你可曾关细化：`lag_refined_by_full_ncc_scan: false`）。
2. 得到全局起点 **`timeline_start_sample = 908404`**（@ 4000 Hz → 约 **227.101 s** 落在全剧时间轴上）。
3. **映射到中文集内时间**（与原片一致，用于同源印尼裁切）  
   - `02.mp4`：**54.205 s – 146.1 s**  
   - `03.mp4`：**0 s – 135.382 s**
4. **印尼目录**下同集、**同一时间码**裁切两段 → **默认重编码分段**（`libx264` + `aac`）→ **`concat` Demuxer + `-c copy`** 合成一条成片。

也就是说：**v2 并不是靠「识别舞曲 vs 观众人声」**，而是靠 **整段爆款的波形** 在原剧连续时间轴上找一个 **整体相关峰**；  
画面能否和爆款「跳舞/观众吐槽」对得上，取决于这个峰**是否真的是同一刀剪辑**，以及 **FFmpeg 入点**是否准。

## 3. 为什么「舞曲」和「观众说话」差很多，程序仍可能对错峰？

互相关 **不做语义分类**：它不区分舞曲、人声、噪声，只算 **整段 needle** 与 hay 上某一段的 **整体相似度**。

若剧里 **同一段 BGM 循环**、或 **能量谱巧合相似**，可能出现 **多个峰**；`argmax` 只认**全局最高的一个**。你看到的 NCC≈**0.47** 本身就说明 **不是「一摸一样」的峰值**，存在 **多峰/模糊** 时选错 **几秒到十几秒** 在工程上是可能的。

所以：**不是「弄错了舞曲和观众」这种分类错误**，而是 **整条里谁和原剧哪一段最像** 的数值游戏；偏低 NCC 时更要人工听一下对齐点。

## 4. 若要复现或「锚定」v2 这一版

- 使用 **同一份** `match_report.json` 中的 `cuts_chinese` 时间码 + 印尼同集路径再导出，**时间轴与 v2 同源**。当前仓库已将 **`--cut-seek-margin` 默认设为 `0`**（单次输入侧 `-ss`），与 v2 成片路径对齐。
- 若个别素材仍觉入点飘，可试 **`--cut-seek-margin 3`**（混合 seek）对照。
