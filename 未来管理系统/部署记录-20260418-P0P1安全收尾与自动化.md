### 2.4 阿里云安全组改造

**改前**：SSH(22) 限定 4 个 IP（家庭 + 3 个 VPN），切 VPN 节点 / 家中重新拨号都会锁出。

**改后**：删除 4 条限 IP 规则，保留一条 `22/tcp from 0.0.0.0/0`，描述 `SSH open (key-only + fail2ban protected)`。理由：应用层已有 fail2ban + 密钥登录两层硬防护。

### 3.3 nginx index.html no-cache

- 修改文件：`/etc/nginx/sites-available/default`
- 备份：`/data/backup/configs/20260418_202141/nginx-default`

新增 location 块（在 `# Static asset cache` 之后、`# SPA fallback` 之前）：

```nginx
# index.html: 永不缓存（保证发版后用户刷新即看到新版）
location = /index.html {
    add_header Cache-Control "no-cache, no-store, must-revalidate" always;
    add_header Pragma "no-cache" always;
    expires 0;
}
```

**效果**：以后前端发版（覆盖 `dist/`），用户**只需刷新浏览器**即可看到新版，不再出现"用户在用半个月前的旧前端"的问题。静态资源（JS/CSS/图片）仍然 `expires 30d, immutable`，HTTP 缓存命中率不受影响。

**响应头验证**：

```

```

## 四、后端发版脚本（带自动回滚）

- 脚本：`/data/scripts/deploy-backend.sh`
- 日志：`/data/logs/deploy-backend.log`
- JAR 备份目录：`/data/backup/jars/`（保留最近 10 份）

### 4.1 用法

```bash
deploy-backend.sh <new-jar-path>   # 部署新 JAR（含自动回滚）
deploy-backend.sh --check          # 只检查环境，不做任何修改
deploy-backend.sh --rollback       # 手动回滚到上一个版本（带二次确认）
deploy-backend.sh --list           # 列出可回滚的备份版本
```

### 4.2 流程

1. 校验环境（JAR/服务/curl/java 都在）
2. 校验新 JAR 文件（存在、可读、> 1MB、PK 头）
3. **备份**当前 JAR 到 `/data/backup/jars/drama-admin_<timestamp>.jar.bak`
4. 替换 `/data/app/drama-admin-0.0.1-SNAPSHOT.jar`
5. `sudo systemctl restart drama-admin`
6. **轮询健康** 最多 90 秒，每 3 秒一次，匹配 `"status":"UP"`
7. **健康** → 部署成功，清理超过 10 份的旧备份
8. **不健康** → **自动回滚**（恢复旧 JAR + 重启 + 再做健康检查），输出最近 30 行 journalctl 帮助定位错因，脚本退出码 2
