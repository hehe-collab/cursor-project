二、根本原因推测
--------------------------------------------------------------------

### 核心问题：Textarea的换行符处理

**素材ID输入框 vs 素材关键词输入框**：

| 特性 | 素材关键词 | 素材ID |
|------|-----------|--------|
| HTML元素 | `<input>` | `<textarea>` |
| 输入内容 | 单行文本 | 多行文本（换行分隔） |
| 换行符 | 不涉及 | `\n` 或回车 |
| Element UI处理 | 简单 | 复杂（验证、格式化） |
| 我们的填入方式 | `fill()` | `fill()` + `\n` |

---

### 为什么第1次只搜到10个？

**可能的原因**：

1. **换行符未被正确识别**
   ```javascript
   // 我们填入
   "7603...\n7603...\n7603...\n7603...\n7603...\n7603...\n7603..."
   
   // Element UI可能识别为
   "7603... 7603... 7603..."  // 换行符变成空格，多个ID合并
   
   // 导致只匹配到部分ID
   ```

2. **Playwright的fill()对textarea不完全兼容**
   - `fill()` 主要为 `<input>` 设计
   - 对 `<textarea>` 的换行符处理可能不正确

3. **Element UI的textarea有特殊验证**
   - 需要实际按Enter键来分隔
   - 或者需要特定的格式触发

---

### 为什么第2次就成功？

**可能的原因**：
- 第1次操作虽然失败，但"激活"了textarea的某些状态
- 第2次填入时，textarea已经"就绪"
- （类似于之前的Vue预热，但这次是textarea本身的预热）
