## 1. GitHub 功能概述

- **代码托管**：仓库、版本控制、提交历史、分支管理
- **协作开发**：Pull Request、Code Review、Issue
- **自动化**：GitHub Actions、CI/CD
- **安全**：Dependabot、私有仓库

---

## 2. 从零开始学习 GitHub

1. 注册账号 → 创建仓库 → 安装 Git → 配置 user.name / user.email
2. `git clone <地址>` → 修改文件 → `git add .` → `git commit -m "说明"` → `git push`

---

## 3. Git 安装详解

- macOS：`git --version` 触发安装 / `brew install git` / 官网 .dmg
- 必配：`git config --global user.name`、`user.email`

---

## 8. git add / commit / push 含义

| 命令 | 作用 |
|------|------|
| `git add .` | 加入暂存区 |
| `git commit -m "说明"` | 本地提交 |
| `git push` | 推送到远程 |

---

## 10. 分支的作用与使用

- 隔离开发、保持 main 稳定、多人协作
- `git checkout -b 分支名`、`git merge 分支名`
- 删除分支 ≠ 撤销合并

---

## 11. 代理配置与网络问题

- Git 需单独配置代理：`git config --global http.https://github.com.proxy http://127.0.0.1:33210`
- 代理端口以代理软件显示为准（如 HTTP:33210）

---

## 12. 分支合并与 git revert

- `git revert`：新建反向提交，不删除历史
- `git revert -m 1 <合并提交ID>` 可撤销合并

## [2026-03-12] 对话记录（素材“像素级”复刻翻刻）

**要点**：
- **结论**：100% 可行，且技术非常成熟。这本质上是一个“视频序列对齐（Video Alignment）”或“基于画面的视频检索”问题。
- **核心思路**：不依赖大模型复杂的语义理解，而是用传统的计算机视觉（CV）或轻量级图像特征模型来进行“找同款”。
- **实现步骤**：
  1. 镜头拆解：将中文素材拆分成独立片段。
  2. 画面裁剪：把中文素材和印尼语剧集的画面底部（字幕区）切掉，消除字幕干扰。
  3. 提取特征：抽取关键帧，计算图像哈希（pHash）或向量特征（如 CLIP）。
  4. 序列匹配：用中文素材的帧特征去剧集库里检索，找到对应的集数和精确时间戳。
  5. 拼接渲染：生成剪辑配置，用 FFmpeg 从印尼剧集中切下对应片段拼接。
- **难点提示**：素材若经过了调色、变速、裁剪（如 16:9 裁成 9:16），或有复杂转场，匹配难度会增加，需要用更鲁棒的特征提取算法。

  - 剪辑师经常为了紧凑节奏，把中文素材的某段画面加速了 1.2 倍到 1.5 倍。

- **复杂转场带来的帧污染（Transitions Blend）**：
  - 中文素材两个镜头切换时，如果有 0.5 秒的“叠化（Crossfade）”或“闪白”，这 0.5 秒内的画面是两张图融合的。
  - **处理策略**：特征匹配在这 0.5 秒极容易失效。系统在做场景切分（Scene Detect）时，需要把转场过渡期的帧“扔掉”，只取镜头最中间、最稳定的画面去检索特征。然后按硬切（Hard Cut）的模式拼起来。短剧本身节奏快，硬切往往比花哨的转场更能吸引注意力。
